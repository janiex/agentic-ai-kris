# Reference: source-quality heuristics

Loaded only when deeper guidance is needed (progressive disclosure, level 3).

## Judging a source

- **Primary > secondary.** Prefer original documentation, specs, and data over
  summaries of them.
- **Recency matters for fast-moving topics.** Note the date of each claim when
  the field changes quickly (tooling, prices, APIs).
- **Corroboration.** A claim supported by two independent passages is stronger
  than one repeated by the same upstream source.
- **Specificity.** Concrete numbers, versions, and named mechanisms are more
  trustworthy than vague generalities.

## Common failure modes to avoid

- Presenting an inference as a retrieved fact.
- Over-generalising from a single example.
- Ignoring a passage that contradicts the preferred answer.
- Burying the actual recommendation under background.

## Citation style

Inline numeric refs tied to the retrieved passages: `[K1]`, `[K2]`. Anything not
traceable to a passage is marked `(unverified)`.
